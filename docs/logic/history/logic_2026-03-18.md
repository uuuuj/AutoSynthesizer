## 변경 1: 500행 초과 데이터 잘라내기 로직 추가

### 변경 위치

- `_analyze_file()` 메서드 내부, `self.original_columns` 설정 직후

### 변경 내용

- 엑셀 파일 로드 후 데이터가 **500행을 초과**하면 상위 500행만 유지하고 나머지 행은 제거
- `messagebox.showinfo`로 사용자에게 안내 메시지 표시 (원본 행 수, 제한 행 수, 제외된 행 수)
- `self.info['rows']`를 잘라낸 행 수로 갱신하여 이후 분석 요약에 정확한 행 수 반영

### 추가된 코드

```python
MAX_ROWS = 500
if len(self.df) > MAX_ROWS:
    original_rows = len(self.df)
    self.df = self.df.iloc[:MAX_ROWS].reset_index(drop=True)
    self.info['rows'] = len(self.df)
    messagebox.showinfo(
        "행 수 제한 안내",
        f"원본 데이터가 {original_rows}행으로 제한({MAX_ROWS}행)을 초과합니다.\n\n"
        f"상위 {MAX_ROWS}행만 사용하고 나머지 {original_rows - MAX_ROWS}행은 제외됩니다.")
```

### 동작 흐름

```
파일 로드 → DataFrame 생성 → 500행 초과 확인
  → 초과 시: 상위 500행만 유지 + 사용자 알림
  → 이하 시: 그대로 진행
→ 컬럼 타입 감지 → 이후 파이프라인 계속
```

## 변경 2: 프로그레스바 동작 안 함 수정

### 원인

- `mode='indeterminate'`(좌우 반복 애니메이션) 사용 중, 워커 스레드에서 `root.after()`로 빈번한 GUI 업데이트를 수행하여 메인 스레드가 바빠지면서 애니메이션이 멈춤

### 변경 내용

- `mode='determinate'` (maximum=100)로 변경하여 실제 진행률 표시
- `_do_synth()` 내부에 프로그레스바 업데이트 헬퍼 `P()` 추가
- 8단계 각 단계 시작 시 프로그레스바 값 업데이트

### 진행률 배분

| 단계 | 프로그레스바 값 |
|------|----------------|
| [1/8] 매핑 수집 | 0% |
| [2/8] 문자열 합성 | 12% |
| [3/8] 종속성 분석 | 25% |
| [4/8] 상관관계 | 37% |
| [5/8] 제약조건 | 50% |
| [6/8] 수치 합성 | 62% |
| [7/8] 행 조합 | 75% |
| [8/8] 품질 검증 | 88% |
| 저장 | 95% |
| 완료 | 100% |

### 주요 변경 코드

```python
# Progressbar 생성 (변경 전 → 후)
# 변경 전: ttk.Progressbar(bf, mode='indeterminate', length=200)
# 변경 후:
self.progress = ttk.Progressbar(bf, mode='determinate', maximum=100, length=200)

# _do_synth() 내 헬퍼 추가
P = lambda v: self.root.after(0, lambda val=v: self.progress.configure(value=val))

# 각 단계마다 P(값) 호출
P(0);  S("[1/8] 매핑 수집...")
P(12); S("[2/8] 문자열 합성...")
# ... (이하 동일)
P(100)  # 완료
```

## 변경 4: ID형 컬럼 가짜 접두사가 원본과 동일한 버그 수정

### 변경 위치

- `_build_input_widgets()` 내 ID형 분기 (line 1474~)
- `_do_synth()` 내 [Issue 4] ID 컬럼 유니크성 보장 (line 1736~)
- 신규 헬퍼 함수: `_extract_id_prefix()`, `_pick_different_prefix()`

### 기존 문제

- ID형 컬럼의 가짜값 접두사가 항상 `'S'`로 하드코딩되어 있었음
- 원본 ID가 `S0001` 형태일 때 가짜값도 `S0001`로 생성되어 매핑이 `원본=가짜` (비식별화 실패)

### 변경 내용

- `_extract_id_prefix()`: ID 값 리스트에서 영문/한글 접두사를 추출
- `_pick_different_prefix()`: 후보 리스트(`X, F, Z, Q, W, J, V, P, R, T`)에서 원본과 다른 접두사 선택
- GUI 표시와 합성 실행 양쪽 모두 원본 접두사와 다른 접두사를 사용하도록 수정

### 예시

| 원본 ID | 변경 전 (버그) | 변경 후 |
|---------|---------------|---------|
| S0001   | S0001         | X0001   |
| A001    | S0001         | X0001   |
| X0001   | S0001         | F0001   |

## 변경 6: 가짜 데이터 자동 생성 시 변환된 컬럼명 기반으로 코드 생성

### 변경 위치

- 신규 헬퍼: `_get_renamed_col()` 메서드
- `_auto_fill_column()` 메서드
- `_collect_mappings()` 메서드 (폴백 코드명 생성 부분)

### 기존 문제

- `generate_auto_codes(col, n, seed)`에 원본 컬럼명을 전달하여 코드가 원본 컬럼명 기반으로 생성됨
- 예: 원본 컬럼 `공정구분` → `공정구_A`, `공정구_B` 형태

### 변경 내용

- `_get_renamed_col(col)`: 컬럼명 변경 Entry에 입력된 새 이름이 있으면 반환, 없으면 원본명 반환
- `_auto_fill_column()`: `generate_auto_codes()`에 변환된 컬럼명(`display_col`)을 전달
- `_collect_mappings()`: 폴백 코드 생성 시에도 변환된 컬럼명 사용

### 예시

| 원본 컬럼명 | 변환 컬럼명 | 변경 전 코드 | 변경 후 코드 |
|------------|------------|-------------|-------------|
| 공정구분 | Text1 | 공정구_A | Text_A |
| 담당부서 | Text2 | 담당부_A | Text_A |
